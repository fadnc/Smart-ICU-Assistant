Here's exactly what changed and why each change matters on an A100:
training.py — the biggest change
The previous file was built for 4 GB VRAM and is now completely replaced. The key upgrades: BF16 replaces FP16 — the A100 has dedicated BF16 tensor cores and doesn't need a GradScaler, which was the root cause of your NaN loss on TCN and Transformer. With BF16 and BCEWithLogitsLoss (instead of BCELoss + external sigmoid), the NaN issue should disappear for Transformer too. Batch size goes from 32 to 2048 — at 980K sequences this means ~480 batches per epoch instead of ~30,600, so each epoch is roughly 60x fewer kernel launches. torch.compile(mode='reduce-overhead') fuses repeated LSTM/Transformer kernels into a CUDA graph, giving another 20–40% speedup after the first epoch's tracing cost (~30s). DataLoader workers go from 2 to 16 with prefetch_factor=4 and persistent_workers=True so the GPU is never waiting on CPU data staging. Fused AdamW (fused=True, foreach=True) runs all parameter updates in a single CUDA kernel instead of one per tensor.
models.py — sigmoid removed from all forward() methods. BCEWithLogitsLoss handles it internally in a numerically stable way that doesn't overflow in BF16. Added LayerNorm after the LSTM output and norm_first=True on the Transformer encoder — both are standard BF16 stability practices. LSTM hidden size widened from 128 → 256, layers 2 → 3, Transformer d_model 128 → 256 since you now have the VRAM headroom.
config.yaml — GPU_CONFIG section entirely rewritten with the A100 values. All comments explain exactly what each setting does and what it was before.
Expected speedup: your current run was doing mortality/LSTM in ~60 minutes per epoch. At batch 2048 + compile + BF16 + 16 workers, expect 3–6 minutes per epoch on the A100 — roughly 10–20x faster end to end.
