Here's exactly what changed and why each change matters on an A100:
training.py — the biggest change
The previous file was built for 4 GB VRAM and is now completely replaced. The key upgrades: BF16 replaces FP16 — the A100 has dedicated BF16 tensor cores and doesn't need a GradScaler, which was the root cause of your NaN loss on TCN and Transformer. With BF16 and BCEWithLogitsLoss (instead of BCELoss + external sigmoid), the NaN issue should disappear for Transformer too. Batch size goes from 32 to 2048 — at 980K sequences this means ~480 batches per epoch instead of ~30,600, so each epoch is roughly 60x fewer kernel launches. torch.compile(mode='reduce-overhead') fuses repeated LSTM/Transformer kernels into a CUDA graph, giving another 20–40% speedup after the first epoch's tracing cost (~30s). DataLoader workers go from 2 to 16 with prefetch_factor=4 and persistent_workers=True so the GPU is never waiting on CPU data staging. Fused AdamW (fused=True, foreach=True) runs all parameter updates in a single CUDA kernel instead of one per tensor.
models.py — sigmoid removed from all forward() methods. BCEWithLogitsLoss handles it internally in a numerically stable way that doesn't overflow in BF16. Added LayerNorm after the LSTM output and norm_first=True on the Transformer encoder — both are standard BF16 stability practices. LSTM hidden size widened from 128 → 256, layers 2 → 3, Transformer d_model 128 → 256 since you now have the VRAM headroom.
config.yaml — GPU_CONFIG section entirely rewritten with the A100 values. All comments explain exactly what each setting does and what it was before.
Expected speedup: your current run was doing mortality/LSTM in ~60 minutes per epoch. At batch 2048 + compile + BF16 + 16 workers, expect 3–6 minutes per epoch on the A100 — roughly 10–20x faster end to end.

Bug 1 — BCELoss crash under AMP (the fatal one)
Your models output raw logits from a final nn.Linear(..., 1) — no sigmoid. BCELoss expects probabilities (0–1), so PyTorch throws. Under FP16 AMP this is also numerically dangerous because sigmoid can saturate to exactly 0.0 or 1.0 in half precision, making log(p) undefined. Fix: replaced BCELoss + MultiTaskLoss with a new MultiTaskBCEWithLogitsLoss that wraps nn.BCEWithLogitsLoss (fused sigmoid + BCE, AMP-safe). The model heads are untouched — they correctly output logits. torch.sigmoid() is now applied only at inference in predict() and in the val loop.
Bug 2 — DataLoader multiprocessing crash on Windows
num_workers=2 in GPU_CONFIG causes PyTorch to spawn worker processes on Windows. This requires the entry point to be guarded by if __name__ == '__main__' — which main_pipeline.py isn't when called from the pipeline. Fix: added _safe_num_workers() which forces num_workers=0 on Windows (detected via platform.system()), falling back to single-process loading. On Linux (your A100 server) it continues using 2 workers as before.
Bug 3 — Deprecated AMP API (warnings)
torch.cuda.amp.GradScaler and torch.cuda.amp.autocast updated to torch.amp.GradScaler('cuda', ...) and torch.amp.autocast('cuda', ...) throughout.

Why GPU was at 46% / 0.2 GB VRAM:
With batch_size=32 and num_workers=0, each training step looked like this: GPU finishes a batch in ~1s → CPU spends ~3s loading the next batch from RAM, converting to tensors, transferring via PCIe → GPU computes again. The GPU was idle 75% of the time. This is a pure CPU-bound bottleneck, not a model problem.
What changed in training.py:
batch_size 32 → 128. Your 4 GB VRAM can comfortably hold an LSTM forward pass at batch=128 in FP16 (uses ~1.8 GB). At batch=32 you were using 5% of your VRAM. Larger batches mean fewer CPU→GPU transfer calls per epoch, so the GPU gets fed faster and stays busy longer.
grad_accum_steps 2 → 1. Gradient accumulation was there to simulate a larger effective batch (32×2=64). With batch=128 it's no longer needed and just adds unnecessary division/scaling per step.
prefetch_factor=2 + persistent_workers=True added for Linux/macOS. On Windows these stay disabled (num_workers is forced to 0), but on your A100 server they'll pre-load batches N+1 and N+2 while the GPU processes batch N, hiding CPU latency entirely.
weights_only=True added to torch.load calls, removing the FutureWarning in PyTorch 2.x.
